# Limira 工作日志

## 2026-06-12 Redis Stream / SSE 可靠性硬化

### 背景

本轮从“生产事故风险”角度检查任务事件链路。当前事件链路是：

```text
runner event
  -> 持久 event log
  -> runtime state
  -> Redis Stream 公共直播频道
  -> 本进程 SSE observer fan-out
```

重点检查五类事故：漏消息、重复消息、乱序、页面卡住、Redis 堆爆。

### 已发现的问题

- 本进程 observer queue 原来是无界队列。慢客户端长期不消费时，后端内存会按消息数增长。
- Redis Stream 读取失败时，跨进程 observer 可能无法区分“直播管道断了”和“任务死了”，容易表现为页面卡住。
- runtime state 更新失败会打断 event emit 路径，Redis 抖动可能影响本进程 SSE 输出。
- SSE 帧没有 `id:`，后续要做 Last-Event-ID 恢复和排查边界时缺少稳定游标。
- 旧的 `stream_already_open` 内部状态必须继续保持对用户不可见。

### 本轮修复

- 给本进程 subscriber queue 增加上限 `LIMIRA_TASK_EVENT_SUBSCRIBER_QUEUE_SIZE = 256`。
- 慢 observer 队列满时从 broadcaster 移除，并插入结束哨兵，避免一个慢用户拖垮任务广播。
- Redis Stream 公共频道继续使用 `MAXLEN = 2000` 和 runtime TTL，避免无限增长和僵尸频道长期保留。
- event emit 改成：持久 event log 尽力写入、runtime state 尽力更新、Redis Stream 尽力追加、本进程广播继续发送。
- Redis Stream 读取失败时返回公开状态事件 `warning: live_stream_reconnecting`，让 EventSource 重连和前端轮询恢复接管。
- SSE 帧增加 `id:` 行；`data:` JSON 不增加内部字段，保持前端兼容。
- 保持 `stream_already_open` 过滤，不进入用户可见 event log。

### 新增测试

- 同进程多 observer：runner 只被调用一次，两个 observer 收到同一条完成事件。
- 跨进程 observer：模拟不同 web 进程，只能通过 Redis Stream 收到 live event。
- Redis Stream 读取失败：observer 收到 `live_stream_reconnecting`，不会无限挂起。
- 慢 observer：超过队列上限会被断开，不会无限堆积。

### 待继续观察

- 当前 Redis Stream 游标仍由后端内部管理；前端尚未显式保存和传回 Last-Event-ID。
- Redis Stream `MAXLEN = 2000` 是固定常量，后续可按部署规模改成环境变量。
- 跨进程恢复依赖 Redis；Redis 长时间不可用时，系统会退回 event log / task status 恢复，但不会提供完整实时过程。

## 2026-06-12 Last-Event-ID / Redis Stream MAXLEN 产品化

### 背景

上一轮留下两个产品化点：

- 前端尚未显式保存并传回 SSE `Last-Event-ID`。
- Redis Stream `MAXLEN = 2000` 仍是代码固定常量。

这两个点直接影响刷新恢复、跨设备观察和部署容量调参。

### 本轮修复

- SSE 每条公开事件继续带 `id:`，Redis Stream 事件使用 `redis:<stream-id>`，event log 回放事件使用 `eventlog:<event_log_id>`。
- 前端按 task 保存最近收到的 `event.lastEventId` 到 workspace localStorage。
- 前端重连同一 task 时通过 `?last_event_id=...` 显式回传 cursor，避免刷新后从错误位置开始。
- 新建任务时清理该 task 的旧 cursor，避免复用历史游标。
- 后端 `/tasks/{task_id}/events` 同时支持 `last_event_id` query 参数和浏览器自动重连时的 `Last-Event-ID` header。
- Redis cursor 恢复时只读取 cursor 之后的 Stream 事件；event log cursor 恢复时只回放 cursor 之后的历史记录。
- 修复 Redis cursor 回放的终态边界：跨进程 observer 由 Redis Stream observer 独占读取，避免 backfill 先吃掉终态事件后主 observer 去重并继续阻塞。
- Redis Stream 长度改为 `LIMIRA_TASK_EVENT_STREAM_MAXLEN` 环境变量，默认仍为 `2000`，部署时可按任务消息量调大或调小。
- `docker-compose.limira.yml` 和 `.env.example` 暴露 `LIMIRA_TASK_EVENT_STREAM_MAXLEN`。

### 新增测试

- Redis Stream observer 可从 `redis:<stream-id>` cursor 后恢复，不重复已见事件，并在终态事件后结束。
- runtime state factory 可读取 `LIMIRA_TASK_EVENT_STREAM_MAXLEN`，非法值会失败，Redis `XADD` 使用配置的 `MAXLEN`。
- 前端合同测试覆盖 `taskEventsUrl()`、`last_event_id` query、`event.lastEventId` 记录和 workspace 持久化。
- 部署合同测试覆盖 compose 和 `.env.example` 中的 `LIMIRA_TASK_EVENT_STREAM_MAXLEN`。

### 验证结果

- `python3 -m py_compile apps/limira-web/backend/limira_backend/routers/limira.py`
- `uv run pytest tests/test_limira_web_routes.py -q -k "last_event_id_cursor or redis_stream_allows_cross_process or multiple_live_observers or runtime_state_factory"`，结果：`4 passed`

## 2026-06-12 重连后 status 噪音 / Final Summary 卡顿排查

### 现场判断

- 用户看到退出重进后工作过程持续出现 `状态 / status`，并多次出现“最终报告已生成”。
- 先查运行日志和任务库，最近任务 `f636e09c-fc57-4f58-839d-bc4873ee5526` 当时仍是 `running / pending`，runner 日志显示已进入 Final Summary，但多次因为 LLM `length` 截断而重试。
- 运行中的 runner、web backend、frontend 进程均启动于 08:20，早于本轮 Last-Event-ID / Redis Stream 修复，因此线上本地服务确实还没加载最新工作树代码。
- 该任务随后在 `2026-06-12T10:22:32+00:00` 自然完成，状态变为 `completed / ready`，归档已生成。

### 已发现的问题

- 前端 `shouldSuppressStatusThinkingStep` 会把 `reason` 等内部字段当成可见详情，导致普通 status/task_update 状态刷新有机会进入工作过程。
- 前端没有对完全相同的工作过程步骤做签名去重，历史回放或旧 SSE 重连会重复展示“最终报告已生成”等步骤。
- runner 重启后，如果 SQLite/Postgres 里留下 `running` 任务但当前进程没有 worker，`GET /tasks/{id}` 和 `GET /events` 过去不会主动收口，需要用户调用 cancel 才会终结。
- Final Summary prompt 对复杂任务仍允许输出过长报告，触发模型 length 重试后会让任务看起来长时间卡在 running。

### 本轮修复

- 前端 status/task_update 只在有 `message`、`summary`、`title`、`warning` 或 `error` 时进入工作过程；纯状态同步保持静默。
- 前端 `appendThinkingStep` 增加 `kind/title/detail/meta` 签名去重，防止重连回放重复堆同一条工作过程。
- runner 的 `GET /tasks/{id}` 和 `GET /events` 在发现任务是 `running` 但本进程没有 active worker 或未完成 worker task 时，自动调用现有归档兜底逻辑终结为 `cancelled`，避免僵尸 running 长期拖住页面。
- worker 存活判断同时检查 active set 和 `TASK_WORKERS`，避免 worker 刚创建但尚未注册 active 的短窗口被误判。
- Final Summary prompt 增加硬长度上限：中文约 5,000 字内，英文约 2,500 词内，优先综合和最强引用，避免继续生成到截断。

## 2026-06-12 Final Summary 硬长度限制撤销 / report.md HTML 卡片泄漏修复

### 现场判断

- 用户指出 Final Summary prompt 不应该靠硬长度上限解决卡顿问题，这个判断正确；长度截断问题应由任务/模型调用稳定性和重试策略处理，不应牺牲最终报告完整性。
- 归档报告中出现 `<div class="search-card">`、`<div class="scrape-card">`、`<div class="tool-card">`，根因是 runner 的 `_render_markdown()` 把所有 agent 的中间搜索、抓取、工具过程先写进 `report.md`，最后再追加 Final Summary。
- 多个旧 archive 的 `report.md` 从第 1 行就是过程 HTML，说明这是归档渲染规则的问题，不是某一次任务输出异常。

### 本轮修复

- 移除 `prompt_patch.py` 中 Final Summary 的硬字数/词数限制，不再要求中文约 5,000 字或英文约 2,500 词内完成。
- 保留“不要输出工具调用、内部推理、搜索/抓取卡片或 HTML UI 片段”的最终回答质量约束。
- 修改 `pipeline_helpers._render_markdown()`：只要存在 `Final Summary` 的 `show_text/message` 正文，归档 `report.md` 就只使用该最终正文。
- Final Summary 阶段里的非文本工具卡片不再进入归档报告。
- 兼容非标准路径：如果 Final Summary 启动但没有正文，而 main agent 已输出一份“像最终报告”的长 `show_text`，则归档改用该最终报告正文，不再回退到过程流。
- 只有在没有 Final Summary 正文时，才降级输出过程流 markdown，便于失败任务排查。
- 渲染错误块从 HTML 改为 Markdown quote，减少归档正文里的 UI HTML。

### 新增测试

- `test_render_markdown_uses_final_summary_without_intermediate_html_cards`：模拟 main agent 产生 search card、Final Summary 产生最终正文，断言最终 markdown 只包含最终报告，不包含 `<div`、`search-card`、`tool-card` 或搜索结果标题。
- `test_render_markdown_uses_main_final_report_when_final_summary_has_no_body`：模拟 main agent 输出完整报告、Final Summary 仅启动但无正文，断言归档仍只保留完整报告正文，不混入搜索卡片。
